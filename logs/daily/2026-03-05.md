## Summary

Bug investigation and fix: autocomplete field_matches (regelingen, begrotingsnamen, artikelen) were not appearing in the dropdown while typing partial words. Root cause: field_matches required exact word boundary matches while ontvangers accepted prefix matches. Fix adds substring fallback matching the same exact+prefix pattern. Also removed public networking on Railway backend (security hardening) and updated local dev `.env.local` to use `localhost:8000`.

---

## Work Completed

1. **Autocomplete field_matches prefix matching fix (V2.4.3)** — Field matches (regelingen, begrotingsnamen, artikelen) in the autocomplete dropdown now appear while the user is still typing partial words. Previously required exact word boundary match ("veehouderij" showed regelingen, but "veeho" didn't). Now uses exact+prefix fallback matching the same pattern as the primary field (ontvangers). Exact word boundary matches prioritized, substring matches fill remaining slots.

2. **Railway backend public networking removed** — Removed the public domain `rijksuitgaven-api-production-3448.up.railway.app` from the backend service. Frontend already uses internal URL `http://rijksuitgaven-api.railway.internal:8080`. Security improvement — backend now only accessible within Railway private network.

3. **Local dev env updated** — Changed `BACKEND_API_URL` in `app/.env.local` from production public URL to `http://localhost:8000`. Required because public networking was removed. Local dev now requires running backend locally.

4. **Bug investigation process** — Full end-to-end trace of autocomplete pipeline: Typesense queries → backend processing → BFF proxy → frontend rendering. Confirmed via production Network tab that the API returns field_matches correctly but `is_word_boundary_match` filter was too strict for partial words.
