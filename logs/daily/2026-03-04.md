## Work Completed

1. **Social template updates** — 4 changes across 3 template sets:
   - Provincie: template 9 "via" → "voor"; added template 10a ("maakte ... over aan")
   - Gemeente: template 11 "keerde uit" → "betaalde"; added template 13a ("maakte ... over aan")
   - Inkoop: added template 26a ("De Rijksoverheid gunde in {jaar} opdrachten aan {leverancier} met een waarde van {bracket}")
   - COA: all 5 templates rewritten with `{jaar}`; fixed template 27 "tussen {bracket}" double-prefix bug (user had "tussen" before bracket which already contains "tussen")

2. **extract_facts.py: jaar added to COA and inkoop** — COA query now groups by `normalize_recipient(ontvanger), jaar` (was just ontvanger). Inkoop query now groups by `normalize_recipient(leverancier), staffel, jaar` (was just leverancier, staffel). Both queries filtered to 2022-2024. CSV headers updated.

3. **Verification system for social posts** — Every template variable used in a post is verified against the source data. Pipeline: (1) choose template, (2) extract `{field_name}` placeholders via `get_template_fields()`, (3) format text with values from source CSV, (4) verify each used value appears in the final text, (5) if ANY field fails → reject post entirely. CSV output: 15 variable columns with "value [Checked & Verified]" when used, empty when not applicable to that module.

4. **Template-aware field verification** — Initial implementation verified ALL fields in the dict, causing 298 instrumenten posts to fail (standard templates don't use `{type}`, but `type` was in the fields dict). Fixed by only verifying fields that the chosen template actually uses. Uses regex extraction of `{...}` placeholders from the template string.

5. **New CSV output format** — `buffer-ready.csv` now has 18 columns: text, module, chars, + 15 template variable columns (ontvanger, bedrag, jaar, descriptor, type, provincie, gemeente, source, begrotingsnaam, kostensoort, leverancier, bracket, categorie, regeling, hashtags). Each variable column shows "value [Checked & Verified]" when used in that post, empty when not applicable.

6. **V2.5 version reshuffle** — VERSIONING.md restructured: V2.5 is now solely "Publieke Deellinks" (was "Zoeken & Vergelijken" with 7 features). Former V2.5 search features moved to V2.6 "Zoeken & Vergelijken". V2.6 Performance→V2.7, V2.7 Data & Integraal→V2.8, V2.8 Polish & Toegankelijkheid→V2.9. Roadmap data regenerated via `copy-roadmap-data.mjs`.

7. **V2.5 architecture planning** — Comprehensive planning session with 6-person virtual team. Documented decisions:
   - **Database:** `shared_links` table with UUID id, 12-char alphanumeric token, JSONB state, view_count, soft delete. RLS policies for user-owned CRUD, service role for public reads.
   - **URL format:** `/s/{token}` — permanent, clean, shareable. 62^12 ≈ 3.2×10^21 possible tokens.
   - **Data flow:** Visitor→BFF (`/api/v1/public/share/[token]`)→Backend (internal, same endpoints, limit=25). No backend changes needed.
   - **Shared view:** Read-only table, 25 rows, frozen search/filters/sort, expandable rows (max 5), no export/pagination/column changes.
   - **Conversion:** Top banner ("Gedeelde weergave"), bottom upsell ("Nog X resultaten"), header CTAs, expand limit message.
   - **Security:** Rate limiting per IP, 50 links/day creation limit, no token expiry by default, referrer policy.
   - **SEO:** Dynamic OG tags (module + search query), `noindex/nofollow`, static branded OG image.
   - **Performance:** BFF-level 5-min cache per token, handles viral scenarios (10K views/hour = 12 backend calls/hour).
   - **Routes:** 5 new routes (POST create, DELETE soft-delete, GET public data, GET public details, page `/s/[token]`). Middleware updated for public path.
   - **Files to create/modify:** 10 files identified in implementation plan.

8. **V2.5 design prototype** — HTML prototype at `docs/designs/v25-shared-view.html`. Editorial data utility aesthetic (Bloomberg meets NRC). IBM Plex Sans + Condensed. Mock data: Instrumenten module with infrastructure recipients. Features: navy share context bar, pink conversion strip, frozen state chips, 18-row data table with expanded ProRail detail, fade-to-upsell gradient, feature checklist. Mobile-responsive.
